GitHub Open Source Contribution: The 10-Step Standard Workflow

Many developers want to contribute to open source but get stuck on the specific Git workflow. Rather than worrying about messing up someone else’s repository, simply master the standard “Forking Workflow.”

Here is the 10-step guide to contributing to any GitHub project.

1. Fork the Repository

Navigate to the target repository (e.g., github.com/remote/test) and click the Fork button in the top-right corner.
This creates a copy of the project under your personal account (github.com/me/test).

2. Clone to Local

Clone your personal copy (the fork) to your local machine.
Note: Do not clone the original author’s repository directly, or you will lack push permissions.

3. Configure Upstream

To track changes from the original repository, you must add it as a remote source, conventionally named upstream.

4. Sync Latest Code

Before writing any code, ensure your local main branch is synchronized with the original repository to avoid conflicts later.

5. Create a Feature Branch

Never commit directly to the main branch. Always create a dedicated branch for your specific task.

6. Code and Stage

Perform your development work. Once finished, add the modified files to the staging area.

7. Commit Changes

Write a clear, descriptive commit message. Adhering to Conventional Commits is highly recommended.

8. Keep Branch Updated (Optional but Recommended)

If your development took several days, the original repository may have moved ahead. Before pushing, it is best practice to rebase your branch against the latest upstream.

9. Push to Origin

Push your feature branch to your personal remote repository (Origin).

10. Open a Pull Request (PR)

  1. Go to your GitHub repository page (github.com/me/test).
  2. GitHub will usually display a banner saying “Compare & pull request”. Click it.
  3. Verify the direction:
    • Base repository: remote/test (main)
    • Head repository: me/test (feat/add-login-module)
  4. Fill in the PR description, reference any relevant Issues (e.g., “Fixes #123”), and click Create pull request.

Common Pitfalls

  • Clean .gitignore: Ensure you are not committing local IDE configurations (like .vscode, .idea) or OS files (.DS_Store).
  • PR Granularity: A PR should do one thing well. Do not combine a bug fix, a new feature, and a code refactor into a single Pull Request. This makes the code review process difficult for maintainers.
