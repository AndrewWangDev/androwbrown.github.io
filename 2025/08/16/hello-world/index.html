Understanding DNS Leaks and Risks in Split-Tunneling Networks

DNS (Domain Name System) is the phonebook of the Internet. However, because it defaults to UDP port 53 and transmits in plaintext, it is notoriously easy to monitor, spoof, or hijack.

The Mechanics of DNS

How Resolution Works

When you request a URL, the browser extracts the domain and follows a strict hierarchy: Browser Cache -> System Hosts -> Network Interface DNS.

If not defined locally, the request goes to the recursive resolver (usually your ISP). If the ISP doesn’t have it cached, it initiates an iterative query to the Root, TLD, and Authoritative Name Servers.

More info: Search in Google

Spoofing and Hijacking

In many network environments, DNS traffic is manipulated in three ways:

  1. ISP Hijacking: Local ISPs often monitor UDP:53 to log user data or reduce inter-network settlement costs.
  2. Middlebox Interference: At international gateways, traffic matching specific blacklists is spoofed, returning invalid IP addresses (DNS Poisoning).
  3. Intranet Auditing: Schools and enterprises often redirect all 53-port traffic to internal audit devices.

More info: Wikipedia: DNS Spoofing

Why “Direct IP Access” Doesn’t Work

If DNS is poisoned, why can’t we just type the IP address directly?

The answer lies in SNI (Server Name Indication) used in HTTPS handshakes.

  • When a client sends a Client Hello packet to establish a TLS connection, it must include the SNI field to tell the server which certificate to present.
  • If you access via a raw IP, the SNI field is empty.
  • Modern servers (especially CDNs) host thousands of domains on a single IP. Without SNI, the server returns a default (incorrect) certificate, causing the handshake to fail with errors like CERT_COMMON_NAME_INVALID.

The Risk of Split-Tunneling

I generally advise against simple split-tunneling strategies for sensitive traffic. Using OpenClash as an example, improper DNS routing often leads to privacy leaks.

OpenClash typically handles DNS requests via a virtual interface. Even with intelligent routing, risks exist.

The “Fake-IP” Fallback Issue

In fake-ip mode, if a request doesn’t match a specific rule, the client has to decide how to route it.

  1. It resolves the domain using both the Remote/Encrypted DNS and the Local/ISP DNS in parallel to determine the fastest route.
  2. If the Local ISP DNS responds faster (which is common for CDNs), the system might accept that IP.
  3. The Leak: If the resolved IP triggers a DIRECT rule, the traffic connects directly through your local gateway. This exposes your request to the local ISP, completely bypassing the encryption tunnel.

Why “IP If Not Match” Isn’t Perfect

Other strategies like Redir-Host (IP If Not Match) also have flaws. They rely on the local DNS to resolve the domain first to check against IP rules. This initial resolution step itself is a DNS Leak—your ISP already knows what domain you are trying to resolve before the proxy client even decides whether to proxy it.

My Solution: The “Clean Separation” Architecture

To fix this, I separate DNS resolution from traffic routing:

  1. OpenWrt acts as the gateway.
  2. AdGuard Home listens on port 53, handling ad-blocking and forwarding upstream queries.
  3. OpenClash (listening on a non-standard port) takes over the upstream DNS over encrypted channels (DoH/DoT), ensuring no fallback to the local ISP.
