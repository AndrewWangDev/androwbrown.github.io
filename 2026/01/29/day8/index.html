Solving the ECH Key Rotation Dilemma for Exclave

In the modern landscape of internet privacy and censorship circumvention, Encrypted Client Hello (ECH) represents the “final frontier” of transport security. While TLS 1.3 encrypts the data payload, the Server Name Indication (SNI)—the destination hostname—has traditionally remained in plaintext during the handshake. This vulnerability allows firewalls (DPI systems) to inspect and block connections based solely on the target domain.

ECH solves this by encrypting the Client Hello message, effectively blinding observers to the destination.

However, operationalizing ECH with robust infrastructure providers like Cloudflare presents a unique engineering challenge for mobile clients, specifically those based on the V2Ray/Xray core, such as Exclave.

Today, I am releasing ECH Auto-Patcher, an open-source tool designed to resolve these compatibility issues.

The Conflict: Security Rotation vs. Client Rigidity

To maintain forward secrecy and prevent replay attacks, Cloudflare rotates its ECH Configs (Public Keys) frequently—often hourly.

For standard browsers (Chrome/Firefox), this rotation is seamless; they utilize DNS-over-HTTPS (DoH) to fetch the latest keys just-in-time. However, for specialized proxy clients like Exclave, particularly when configured with advanced transport protocols (mTLS, Multiplexing, XUDP), the situation is different:

  1. Static Configuration: These clients often rely on rigid binary configurations (Protobuf) to support complex features. They lack the native logic to “hot-swap” ECH keys derived from DNS without breaking other transport settings.
  2. Manual Toil: Users are forced to manually look up the new DNS record and update their client configuration every time the key rotates. If the key expires, the connection drops.

Why Client-Side DoH is Suboptimal

The mainstream workaround involves forcing the client to perform a DoH lookup before every connection. While functional, this introduces significant latency overhead:

  • Standard Connection: TCP Handshake (1 RTT) + TLS Handshake (1 RTT) = 2 RTT.
  • With Client-Side DoH: DoH Connection (2 RTT) + DNS Query (1 RTT) + Target Connection (2 RTT) = 5+ RTTs.

On mobile networks, this “Cold Start” penalty can add 200ms-500ms to every request. Furthermore, public DoH endpoints are frequently blocked in restrictive network environments, adding another point of failure.

The Solution: Server-Side Binary Patching

ECH Auto-Patcher shifts the complexity from the resource-constrained client to the server (VPS). It functions as a lightweight middleware daemon.

How It Works

  1. Polling: The script periodically queries Cloudflare’s DNS to fetch the latest ECH configuration.
  2. Binary Surgery: It utilizes a Fixed-Length Binary Replacement technique.
    • The Trick: We pre-allocate a “placeholder” (consisting of 1000 ‘a’ characters) within the Exclave Protobuf backup file.
    • The script locates this placeholder and injects the fresh ECH key into the binary stream, padding the remainder with spaces.
    • This ensures the file structure and offsets remain identical, preserving delicate settings like mTLS certificates and Mux configurations without needing to decode/re-encode the entire Protobuf structure.
  3. Serving: The patched, always-fresh configuration is served via a local subscription link protected by the proxy’s internal loopback interface.

The Result

By moving the update logic to the backend, we achieve:

  • Zero-Added Latency: The client receives a pre-validated static config. Connections establish instantly with standard TLS 1.3 performance.
  • High Availability: Key fetching happens on the VPS, bypassing local network restrictions or DoH blocks.
  • Privacy Hardening: Full ECH support is enabled without sacrificing mTLS or XUDP performance.

The project is open-source and available on GitHub.
